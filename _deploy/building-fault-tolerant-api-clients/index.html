Building Fault-tolerant API Clients

API

One of the crucial aspects in consuming a webservice is fault tolerance, the ability to handle application/network/unpredictable errors and provide graceful degradation to the software’s flow.

By default, developers tend to like extensive documentation that gets outdated in a moment, since they think they are protected against integration errors thanks to the fact that the parts involved in the integration share the same knowledge about the messaging interface.

This is a good approach, since we share a common interface and agree on that kind of contract, but what if we need to change the interface due to an evolution of the domain?
